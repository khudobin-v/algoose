# Задача: Фильтрующий итератор

> **Уровень сложности:** Easy-Medium  
> **Тема:** Паттерны проектирования, итераторы  
> **Встречается на:** Яндекс, Google, Microsoft

---

## 1. Что просят?

```
Реализовать итератор, который оборачивает существующий поток данных
и возвращает только чётные числа.

Интерфейс:
  hasNext() → bool
  next()    → int

Пример:
  iter([1,2,3,4,5,6]):
    hasNext() → true
    next()    → 2
    next()    → 4
    next()    → 6
    hasNext() → false
```

---

## 2. Ключевая идея

Хранить "предзагруженный" следующий чётный элемент. При создании и после каждого `next()` — вызывать `_advance()`, который ищет следующее чётное число.

---

## 3. Пошаговый алгоритм

```
При создании:
  _advance()  // находим первое чётное

_advance():
  двигаемся по source пока не найдём чётное
  если нашли: сохраняем в _nextEven
  если исчерпали: _nextEven = null

hasNext(): return _nextEven != null

next():
  val = _nextEven
  _advance()   // готовимся к следующему вызову
  return val
```

---

## 4. Трассировка

**[1,2,3,4,5]**

```
Создание: _advance()
  source→1 (нечётный), source→2 (чётный!) → _nextEven=2

hasNext() → true
next():
  val=2, _advance(): source→3(нет), source→4(да!) → _nextEven=4
  return 2

next():
  val=4, _advance(): source→5(нет), конец → _nextEven=null
  return 4

hasNext() → false
```

---

## 5. Почему предзагрузка?

`hasNext()` должен быть детерминированным — нельзя "смотреть вперёд" только при вызове `next()`. Предзагрузка гарантирует что `hasNext()` всегда отвечает правильно.

---

## 6. Сложность

| | Сложность |
|---|---|
| `next()` | O(k) — k нечётных элементов пропускаем |
| `hasNext()` | O(1) |
| Суммарно | O(n) |

---

## 7. Типичные ошибки

**Ошибка 1: Искать следующее чётное только в `next()`, игнорируя `hasNext()`.**
`hasNext()` должен работать без продвижения позиции.

**Ошибка 2: Вызывать `_advance()` в `hasNext()`.**
`_advance()` меняет состояние — нельзя вызывать в "чистом" методе проверки.
